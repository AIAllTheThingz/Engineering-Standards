## Purpose

This guide defines how a downstream repository adopts the engineering standards repository without weakening organization controls. Adoption is complete only when the repository has a valid manifest, valid governance configuration, applicable agent standards, reusable governance workflow, branch protection, completion evidence, and an owner who is accountable for ongoing drift remediation.

The adoption process is intentionally staged. Teams MAY begin in advisory mode to discover gaps, but they MUST move to enforcement before relying on the repository for production work, regulated data, privileged infrastructure, deployment automation, or security-sensitive code.

## Applicability

Use this guide for new services, existing services, libraries, infrastructure repositories, database projects, worker services, frontend applications, internal tools, and repositories that primarily hold operational documentation. A repository is in scope when it contains source code, deployment configuration, test assets, automation, security decisions, data-handling rules, or instructions that agents or maintainers use to change production-facing systems.

Repositories that only mirror generated artifacts MAY adopt a reduced profile, but the reduction MUST be recorded in `governance.config.json`, approved through the exception process when mandatory controls are disabled, and validated by repository health checks.

## Adoption Roles

Every adopting repository MUST name accountable owners in `project-manifest.json`. The owner set must include a technical maintainer who can review implementation changes, a CI maintainer who can repair workflow failures, and an escalation contact for exceptions or production-impacting findings.

Reviewers are responsible for verifying that the repository did not merely copy files. They MUST confirm that manifest fields reflect the actual system, workflow permissions are least privilege, evidence is generated by real commands, and branch protection requires the governance checks that the repository claims to enforce.

## Prerequisites

Before adoption, confirm that maintainers have permission to edit repository settings, create or update workflows, configure branch protection, and review security-sensitive findings. Local validation requires PowerShell 7, Git, and the runtime tools for the project type, such as .NET, Node.js, database migration tooling, or infrastructure CLIs.

The repository MUST have no known unresolved secret exposure before governance enforcement is enabled. If adoption discovers exposed credentials, stop normal onboarding, rotate the credential, remove the exposure from history according to the organization incident process, and attach incident evidence separately from normal completion evidence.

## Inventory

Begin with a repository inventory. Identify current CI workflows, required build commands, deployment paths, owners, CODEOWNERS coverage, secrets providers, production environments, data classifications, external integrations, package ecosystems, documentation, local agent instructions, and known exceptions.

The inventory SHOULD produce a short adoption note in the pull request summary. The note must state whether the repository handles restricted data, changes production infrastructure, performs authentication or authorization, manages cryptography, or executes destructive operations. These factors drive risk classification and review depth.

## Risk Classification

Classify the repository using `governance/RISK_CLASSIFICATION.md` before selecting workflow enforcement. Low-risk repositories still need evidence, but High and Critical repositories require stronger review, explicit rollback planning, approval evidence, and stricter branch protection.

Risk classification MUST be based on actual behavior, not repository size. A small script that rotates production secrets, runs migrations, or changes cloud permissions is high risk even when the codebase is small.

## Required Files

An adopting repository MUST contain `README.md`, `SECURITY.md`, `CONTRIBUTING.md`, `AGENTS.md`, `project-manifest.json`, `governance.config.json`, an evidence directory, and at least one governance workflow. Repositories with technology-specific requirements MUST also reference the applicable central agent standard in `governance.config.json`.

The manifest describes the system. The governance configuration describes how controls are validated. The local `AGENTS.md` describes repository-specific instructions and MUST NOT weaken central standards.

## Manifest Setup

Create `project-manifest.json` from the repository template, then replace every example value with repository-specific facts. For schema `1.2.0`, set `governanceVersion` to the semantic release, `governanceCommitSha` to the exact workflow pin, `workflowInterfaceVersion` to `1.0.0`, and declare structured owners, standards consumption, local/hosted evidence, and structured exceptions.

Owners MUST be real people or teams with stable GitHub identifiers, substantive responsibility, and escalation. Email contacts are informational and cannot be the only enforceable owner. Evidence declarations MUST distinguish caller-local files from hosted workflow artifacts. Exceptions MUST be active structured `GOV-*` records and must not be used as a general backlog marker.

## Configuration Setup

Create `governance.config.json` from the template and list every required documentation path, applicable agent standard, validation category, allowlist record, and disabled control. The config MUST match the repository layout. A repository that references a central standard that does not apply to its technology creates review noise and should be corrected during adoption.

Allowlist entries MUST include a path, pattern or rule id, owner, reason, and expiration. Expired allowlists are treated as validation failures or warnings according to the scanner severity model.

## Agent Standards

The root `AGENTS.md` must inherit `agents/AGENTS_Base.md` and any technology-specific standard that applies to the repository. Local instructions can add commands, domain constraints, ownership notes, data-handling rules, and architecture boundaries.

Local instructions MUST NOT permit unsafe behavior that the central standards forbid. If local operational reality conflicts with a central rule, open an exception request before merging adoption.

## Workflow Adoption

Use the reusable governance workflow from this repository and pin it to an immutable commit SHA for production enforcement. During initial adoption, a protected release tag may be used while the team prepares a SHA pin, but SHA pinning is the preferred control.

Example downstream invocation:

```yaml
name: Governance

on:
  pull_request:
  push:
    branches: [master, main]

permissions:
  contents: read

jobs:
  governance:
    uses: AIAllTheThingz/Engineering-Standards/.github/workflows/governance-ci-reusable.yml@<pinned-commit-sha>
    with:
      project-path: .
      governance-version: 1.1.0
      artifact-retention-days: 30
```

The workflow MUST run on pull requests and on pushes to protected long-lived branches. It uses only `contents: read` and uploads evidence with explicit retention. `project-path` is relative to the caller checkout and rejects absolute paths, traversal, and workspace escapes. Every symbolic link, junction, or reparse point anywhere in caller content is unsupported, even when its target stays inside the checkout. This conservative fail-closed rule prevents workspace-boundary and validator-confusion attacks. `governance-version` must match the caller manifest, and `artifact-retention-days` must remain in the supported range.

The job uses three sibling workspaces: `caller/` for the exact caller repository and `${{ github.sha }}`, `standards/` for the Engineering Standards repository and immutable `${{ job.workflow_sha }}`, and `evidence/` for generated reports. Validators execute only from `standards/` and receive the resolved caller root explicitly. The caller cannot override the standards repository or SHA. Evidence records both identities and selects the caller risk classification from the validated manifest.

Downstream validation always includes the governance contract. Additional safe static checks are selected from validated `governance.config.json` categories. Repository-owned builds, Pester suites, scripts, and examples remain the responsibility of the caller's own least-privilege CI; the reusable governance workflow does not execute them. Engineering Standards maintainer tests and examples run only when the caller is this governance repository.

GitHub Enterprise Server does not currently expose the `job.workflow_repository`, `job.workflow_sha`, and `job.workflow_ref` properties used for the trusted checkout. The workflow therefore fails closed on GHES and has no branch, tag, or caller-input fallback.

Migration: remove the former `run-examples`, `run-pester`, and `run-documentation-validation` inputs from callers. They previously could not disable their corresponding checks and are no longer part of the interface. Keep project-specific checks in caller-owned jobs and use `governance.config.json` only for supported central static validation categories.

The supported reusable workflow path is `.github/workflows/governance-ci-reusable.yml`. Files under the root `workflows/` directory are distribution templates and are not directly executable reusable workflows from their current location.

The public `AIAllTheThingz/Engineering-Standards-Canary` repository demonstrates the supported minimal cross-repository integration and verifies release candidates across success and four isolated failure paths. See [Downstream Governance Canary](DOWNSTREAM_CANARY.md). Consumers may compare their workflow boundary and configuration with that canary, but must still run their own builds, tests, security checks, and deployment validation.

## Advisory Mode

Advisory mode is used to discover gaps before enforcement. Findings in advisory mode MUST be triaged into remediation, approved exception, or documented non-applicability. Advisory mode is time-bound; High and Critical repositories SHOULD define an enforcement date before merging the adoption branch.

Advisory results are still evidence. Do not report an advisory failure as Passed. Use `Failed`, `Blocked`, `NotRun`, or `NotApplicable` honestly.

## Enforcement Mode

Enforcement mode begins when branch protection requires the governance workflow, repository health, documentation completeness, schema validation, forbidden-pattern scanning, and technology-specific checks. Pull requests that fail mandatory checks MUST NOT merge unless an emergency exception is approved.

When enforcement is enabled, the repository owner MUST monitor failures and governance drift. A governance workflow that stops running is itself a failure.

## Evidence

Adoption evidence MUST include the manifest, configuration, workflow run, validation results, branch protection settings or screenshot reference, and any approved exceptions. Completion evidence must be generated after validation runs, not before.

Evidence must distinguish automated validation from manual review. Manual validation is acceptable only when the reviewer, method, scope, and limitations are recorded.

## Branch Protection

Apply `docs/BRANCH_PROTECTION.md` to `master`, `main`, and any long-lived release branches. Require pull requests, CODEOWNERS review, conversation resolution, required checks, restricted force pushes, and deletion protection.

If the repository still uses only `master` or only `main`, configure the branch that actually receives production changes and record the branch policy in the adoption evidence.

## Migration From Copied Standards

Copied standards SHOULD be replaced by central references. If a local copy is required for tooling or network isolation, record the source, version, update owner, drift-detection method, and the reason a local copy is necessary.

A copied document becomes stale quickly. Treat it as an operational risk and review it during every governance release.

## Exceptions

Exceptions MUST follow `governance/EXCEPTION_PROCESS.md`. Each exception requires a `GOV-*` identifier, owner, approval, expiration, compensating control, risk acceptance, and remediation path.

Adoption MUST NOT use exceptions to hide missing ownership, absent evidence, disabled branch protection, fake test commands, or known secret exposure.

## Validation

Run the local governance validation before requesting review:

```powershell
pwsh -NoProfile -File scripts/Test-YamlSyntax.ps1 -Path .
pwsh -NoProfile -File scripts/Test-GitHubWorkflowArchitecture.ps1 -Path .
pwsh -NoProfile -File scripts/Invoke-GovernanceValidation.ps1 -Path . -Category JsonSchemas,YamlSyntax,WorkflowArchitecture,MarkdownLinks,DocumentationCompleteness,Contract,ForbiddenPatterns,RepositoryHealth,Evidence,Examples
```

For downstream repositories, run the reusable workflow in GitHub Actions and attach the uploaded governance evidence artifact. Technology repositories MUST also run their build, lint, test, migration, security, or deployment dry-run checks.

## Failure Behavior

If adoption validation fails, fix the root cause before enabling enforcement. If the failure is caused by missing tooling, record `NotRun` or `Blocked`; do not mark the result Passed. If the failure blocks a critical production repair, follow the emergency exception path and include compensating controls.

Expired exceptions, missing owners, missing evidence, unpinned workflows, and disabled mandatory controls are adoption blockers.
